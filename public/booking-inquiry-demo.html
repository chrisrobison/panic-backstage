panic-booking-inquiry · public/assets/panic-booking-inquiry.js

A booking form your website can host, that we triage.

Drop one custom element and one script tag into any page — your own marketing site, a promoter's landing page, anywhere — and inquiries land straight in Backstage's Leads pipeline, ready to triage, evaluate, and convert into a real event.

No API key, no backend of your own to run. The component posts directly to Backstage's public inquiry endpoint, which is open to any origin the same way a "contact us" form would be.

03What happens on the server

The component posts JSON to POST /api/public/inquiries, handled by src/PublicInquiry.php — the one write endpoint in Backstage built to accept unauthenticated, cross-origin requests.

Where it landsA row in the same leads table the internal Leads pipeline reads (source = website, status = new). There's no separate inbox — it shows up in Leads like any other inquiry, ready to triage, evaluate, and convert to an event.
CORSWide open (Access-Control-Allow-Origin: *) — the endpoint answers its own preflight OPTIONS request. No cookies or credentials are read or sent, so this is the same trust model as any public "contact us" form.
NotificationsVenue admins with email notifications enabled get an email the moment a lead comes in, linking straight into the Leads pipeline.
Anti-spamA honeypot field (invisible to people, irresistible to bots that fill every input), plus a per-IP and per-email rate limit. Neither is bulletproof, but both raise the bar past a copy-pasted script.
Validationcontact_name, a valid contact_email, and message are required; everything else is optional. Invalid submissions get a 422 with an error message the component surfaces inline.

See docs/openapi.yaml (path /public/inquiries) for the full request/response schema, and README.md for how this sits alongside Backstage's other unauthenticated endpoints (public event pages, ticket checkout, the events feed that powers <mab-events-carousel>).
